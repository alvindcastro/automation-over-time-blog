---
title: "Phase 2: The Better Way"
description: "Why I moved to a structure that stays green."
pubDate: 2025-10-15
---

I threw out the recordings and started over with the Page Object Model.

The idea behind POM is straightforward: instead of writing tests that directly click on elements, you create a class that represents a page or a form. That class knows how to interact with its own elements. The test just calls methods on that class. If a button moves or a label changes, you fix it in the class — once — and every test that uses that form keeps working.

For this system, that meant writing one class per form. Each one was a wrapper around a specific screen, exposing the actions a user would actually take: click Go, fill in an ID, move to the next section.

There was an extra layer of complexity specific to this application. The entire system runs inside an iframe. Every selector had to go through a frame locator first before it could reach any element on the screen. That's a detail that recordings handle invisibly but that becomes very deliberate when you're writing page objects by hand. It forced me to think clearly about what I was actually targeting and why.

I also ran into a subtle problem with autocomplete fields. Typing into them too quickly meant the autocomplete dropdown never had a chance to fire. I had to slow the input down deliberately — character by character with a small delay — to match the pace the system expected. That kind of thing doesn't show up in a tutorial. You find it by watching your tests fail in a specific, repeatable way.

The other decision I made early was to use ARIA-based selectors instead of positional ones like CSS paths or XPath. The application's markup is generated by a legacy stack, and its structure shifts. ARIA roles and labels are far more stable than positional selectors or generated class names. If the form is accessible, the selector holds.

Writing this way was slower. The first few form classes took real effort. But once they existed, they didn't break on their own. A change in the application's UI was now a one-line fix, not an afternoon of chasing cascading failures.

That stability was the only thing I cared about at this point.
